附录：飞书能力全景 ​

飞书这侧的技术底座。支撑 Q2–Q4 的"飞书到底能干什么、边界在哪"，不靠猜、不吹牛。

核心结论 ​

第一版那套「素材库 + 分发回收 + 手动回收」，用飞书原生能力就能搭起来，不需要自己写后端。真正要注意的不是"能不能做"，是免费额度、权限开通、AI 不越权这三条边界。

飞书多维表格（Bitable）：第一版的数据底座 ​

本质是"带界面的数据库" ​

多维表格每列是有类型的字段，每行是一条记录，天然适合当轻量数据库用。第一版两张表（素材资产库、分发回收表）直接建在这里，不用写后端。

对素材库的字段映射 ​

素材库要存的东西字段类型
素材编号自动编号 或 文本（公式生成规范编号）
封面图附件（支持画廊视图预览）
视频链接超链接
卖点/人群/场景多选（可筛选、可统计）
合规状态单选（待确认 / 已确认 / 有风险）
来源/授权状态单选（自有拍摄 / 达人授权可分发 / 仅限自营 / 需重新授权）
成交/咨询数据数字
负责人人员
入库时间创建时间（自动）

内置 AI 捷径：打标签不用写外部脚本 ​

多维表格自带"AI 捷径"，能基于其他列内容自动填充某个字段。第一版可以直接用飞书内置 AI 干：

  • 读素材标题/描述，自动打卖点、人群、场景标签
  • 给素材生成一句话摘要
  • 把口播稿改写成适合代理直接用的短标题

复杂的（结合 SKU 事实包 + 合规规则的生成），再走外部脚本或 CLI。

多视图：同一份数据，不同人看不同视角 ​

同一张表可切换：表格、看板、画廊（封面墙）、甘特、日历、表单。代理回传使用反馈可用表单视图，不用给他们开整张表的权限。

内置自动化：低频提醒可以，别拿它当高频引擎 ​

触发器支持：新增记录、记录满足条件、字段更新、定时、Webhook。动作支持：更新记录、发飞书消息、@负责人、调用 Webhook。

边界：免费版 200 次/月，超出要付费。第一版用场景：日报定时推、新增异常记录时通知负责人。不要拿它跑高频自动化。

开放平台 REST API：脚本写回的正经路子 ​

飞书多维表格有完整的开放平台 REST API，Python SDK 官方维护，稳定。批量写入单次上限 500 行，超过分批。

权限申请流程（这是 Q3 里"Day 1 踢出去"的那件事）：

  1. 飞书管理后台 → 应用管理 → 创建企业自建应用
  2. 申请 bitable:app 权限（写入多维表格的关键权限）
  3. 管理员审批通过后才能用

权限审批时间不在我手上，所以 Q3 Day 1 就提交申请，等待期间先手动占坑，权限到位后脚本自动补。

典型写回场景（Q4 脚本的核心）：

python
# 伪代码示意
import lark_oapi as lark

# 1. 初始化 client
client = lark.Client.builder().app_id(APP_ID).app_secret(APP_SECRET).build()

# 2. 读取清洗好的 Excel
records = parse_excel("千川导出_20260709.xlsx")

# 3. 用素材 ID 作主键，批量 upsert（匹配失败进异常队列）
for batch in chunks(records, 500):
    client.bitable.v1.app_table_record.batch_update(...)

lark-cli：开源 CLI 工具（加分项） ​

2026 年 3 月开源，可用命令行操作多维表格、文档等资源。对第一版是加分项，不是主路径——主路径是上面那套 REST API + Python SDK。

如果你在演示时提到 lark-cli，要清楚它的定位：方便本地调试和快速操作，不替代生产级的 SDK 调用。

官方 MCP（模型上下文协议） ​

飞书官方提供了 MCP 接入，支持 AI 助手直接读写飞书资源（文档、表格、日历等）。

用途：让 Claude 等 AI 工具直接查询素材库里的数据，不用手动导出。当前版本（2026-07）处于早期阶段，第一版方案里定位为技术加分项，不作为核心依赖。

边界与避坑 ​

边界说明
免费自动化 200 次/月超出要付费，第一版在这个额度内设计
权限审批有等待期Day 1 提申请，Day 4 才有可能用上
AI 不做最终确认合规、价格、发布动作必须人工确认
抖音侧有开放平台 API但需要服务商资质，新人短期用不上；实操 = 导出 Excel + 浏览器插件
视频不要存附件字段走超链接字段存播放地址，避免超出附件大小限制
