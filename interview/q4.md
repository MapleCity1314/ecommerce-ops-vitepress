# Q4 · 后台没 API、只能导出 Excel，怎么办？

<p class="hqq-lede">撞到"没有接口"这个约束，第一反应是等，还是换条路？判断比工具更重要。</p>

## 结论

**没 API 照跑，这对第一版不是障碍。**

我要的不是实时数据流，是每天一次、口径固定、能对上账的回收数据进库。导出 Excel 就能满足，一行研发排期都不用等。

---

## 把两侧分清：取数在抖音侧，入库在飞书侧

**取数（抖音侧）：** 没有接口，现实入口就是人工导出 Excel。要自动化，路子不是等接口，是浏览器插件——跑在已登录的浏览器里，一键导出，不用资质、不用模拟登录。**但第一版连插件都先不搭**，就让人点一下导出。先把闭环跑通证明有用，再决定值不值得投自动化。

**入库（飞书侧）：** 库建在飞书，飞书有全套 API 且立即可用，脚本把清洗好的数据批量写回表里，这一段稳、全自动。

---

## 七步，前三步是关键

**第一步：主键先定，用素材 ID，不用计划名。**

这是整件事的地基。千川报表里有创意/素材 ID，这是最稳的主键——一条素材可能挂在多个计划下，用计划名做键会漏掉素材维度的归并，而且批量托管的计划名未必让投手逐条自定义。主键落在素材 ID 上，计划名里的实验编号（格式定死，如 `HQ77-20260709-A1`）只作辅助校验。这一步在投放前约定好，命名对了后面全自动，命名乱了脚本也拼不回来。

**第二步：每天固定时间导出 Excel。**

谁导、几点导、导哪几张报表，写进 SOP。文件名按日期和平台命名，不靠人脑记。

**第三步：脚本清洗 Excel。**

统一列名、时间格式、字段口径。平台改一次导出格式，只改这一个清洗脚本，别处不动。

**第四步之后（全是脚本）：**
- 用素材 ID 当主键，回写飞书表（走 API 批量写）
- 匹配失败进异常队列，绝不自动覆盖——宁可漏、不可脏
- 每次导入留原始文件 + 导入日志，随时能查

整条链人只碰两个点：**定命名（一次）、点导出（每天一下）**。

---

## 取数自动化的路径（分级，按值不值得往上走）

```
第一级：人点导出（第一版，稳定，免费）
   ↓ 导出频次高、手扛不住了
第二级：浏览器插件（Chrome extension）
         跑在已登录浏览器，不要资质，比无头 RPA 稳
   ↓ 插件覆盖不了的场景
第三级：无头 RPA（成本中，需维护）
   ↓ 业务真正做大，值得投
第四级：服务商资质 + 开放平台 API（最强，要资质审批周期）
```

浏览器插件这条路我亲测过——飞书文档都能一键导成 markdown，千川后台导 Excel 同理。不是纸上谈兵。

---

## 诚实边界

没接口就没实时性。第一版是每天一次的节奏，不是实时大屏，这是硬约束。实时要等资质接口，我不吹这个第一周就能有。

命名规范是这套的命门，靠人执行。我会把它做进投放计划模板（能选就不手打）+ 每日入库前的命名合规校验（不合规当场退回投放的人改）+ 异常队列。用流程兜人性，不指望人自觉。
