# Q6 · 方案里哪些真需要 Agent，哪些规则脚本就够？

<p class="hqq-lede">这题最能暴露"熟不熟"。老板要测的不是你会用 Agent，是你知道什么时候不用。</p>

## 结论

**默认脚本，Agent 是例外。** 绝大多数活儿，规则和脚本就够了；Agent 只留给"输入没结构、规则写不出来、必须靠语义判断"的少数地方。

---

## 一条铁律

> **这件事的逻辑，我能不能用规则明确写出来？**
>
> - 能写出来 → 脚本。确定、便宜、不会错、出了问题一眼能定位。
> - 写不出来，输入是人话/图片/杂乱信息 → 才上 Agent。

为什么要卡这么死？Agent 有三个脚本没有的毛病：会错（幻觉）、贵（每次调用烧钱）、难查（错了不好定位）。一件脚本能干的事，你上了 Agent，等于花更多钱、买更多不确定性，还更难维护。这不是先进，是浪费。

---

## 规则脚本就够的（方案里的绝大多数）

这些活儿输入有结构、逻辑能穷举，碰它们用 Agent 就是浪费：

| 场景 | 为什么脚本够 |
|---|---|
| 字段映射（千川不同报表里"消耗"和"花费"、"成交"和"转化"对齐） | 映射关系固定，一张表定义一次，脚本机械套用 |
| 去重、入库、增删改查、实时更新 | 数据库基本功，要的是"永远不出错"，不是"聪明" |
| 实验编号生成、命名合规校验 | 规则可穷举，脚本又快又稳 |
| Excel 清洗回写（见 Q4） | 字段清洗是确定逻辑 |
| 异常报警（CTR 高但 3 秒留存低） | 阈值判断，规则就能触发 |

方案的骨架这一整块恰恰**不该**碰 Agent——这些地方要的是"永远不出错"，不是聪明。

---

## 真需要 Agent 的（少数，但价值高）

这些地方输入非结构化，规则覆盖不全，必须靠语义理解：

**客服原话 / 竞品评论聚类**：一堆散乱的用户说法，要归出"购买理由是什么、质疑点在哪、差评苗头是什么"。这没法用规则写，得靠模型读懂语义再归类。

**素材候选生成**：结合 SKU 事实、平台语气、人群场景、合规限制，生成封面/钩子/详情页候选。规则写不出一条好文案。（这正是技术面那个原型干的活。）

**复盘归因草稿**：把曝光、点击、3 秒留存、成交、评论一起摆上来，写出"这条大概率是封面承诺和开头不一致，还是承接页价格问题"。多因素、要解释、要下判断，规则只能报警，做不了归因。

**规则要区分清楚**：异常检测本身是脚本（阈值报警）；解释这个异常**为什么**发生，才是 Agent。报警和归因，一个脚本一个 Agent，别混。

---

## 我自己否掉的那个"聪明想法"

我一度设计过一套方案：把员工填报全交给 Agent，Agent 接管浏览器、录制回放操作、顺手把数据发回来；甚至做了多智能体共享会话缓存，想省掉给每个平台定制的成本。听着很无感很先进。

但我自己掂量下来否了，原因很实在：

- 上下文会混乱，多个人的活儿硬塞进共享上下文，有概率串味。
- 鸡蛋全在一个篮子，一处出问题整条线全塌。企业方案最忌讳这个。
- 贵，而且要保证不出错的代价不成比例。

最后把方案收敛成：填报这件事根本不用 Agent，员工填一张表单，程序做字段映射、一键分发、入库回测，全是普通程序员基本功，安全、便宜、快。

这段经历就是这道题的答案本身：我不是不会上 Agent，是我试过把它上到不该上的地方，然后自己把它拿下来了。**知道 Agent 的边界，比会用 Agent 更值钱。**

---

## 诚实边界

Agent 那几块（聚类、生成、归因）出的都是草稿，要人审——归因草稿可能编因果，素材候选可能踩合规。方案里这些地方全带人审边界，不是 AI 说了算。

"规则能不能写出来"这条线实操里有灰度，我不教条，一件一件看。那个共享缓存的想法我留着当自己的实验，但清楚它的定位，不往生产方案里塞。
