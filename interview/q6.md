# Q6 · 方案里哪些真需要 Agent，哪些规则脚本就够？

<p class="hqq-lede">知道什么时候不该用 Agent，比会用 Agent 更值钱。克制是判断力，不是保守。</p>

## 结论

**默认脚本，Agent 是例外。** 绝大多数活儿，规则和脚本就够了；Agent 只留给"输入没结构、规则写不出来、必须靠语义判断"的少数地方。

## 一个被否决的设想

曾经设计过一套方案：把员工填报全交给 Agent，Agent 接管浏览器、录制回放操作、顺手把数据发回来；甚至做了多智能体共享会话缓存，想省掉给每个平台定制的成本。表面上先进。

但经过实际评估后否决，原因很实在：

- 上下文会混乱，多个人的活儿硬塞进共享上下文，有概率串味。
- 鸡蛋全在一个篮子，一处出问题整条线全塌。企业方案最忌讳这个。
- 贵，而且要保证不出错的代价不成比例。

最后把方案收敛成：填报这件事根本不用 Agent，员工填一张表单，程序做字段映射、一键分发、入库回测，全是普通程序员基本功，安全、便宜、快。

这段经历就是这道题的答案本身：不是不会上 Agent，是试过把它上到不该上的地方，然后自己把它拿下来了。**知道 Agent 的边界，比会用 Agent 更值钱。**

---

## 诚实边界

Agent 那几块（聚类、生成、归因）出的都是草稿，要人审——归因草稿可能编因果，素材候选可能踩合规。方案里这些地方全带人审边界，不是 AI 说了算。

"规则能不能写出来"这条线实操里有灰度，不教条，一件一件看。那个共享缓存的想法保留当实验，但清楚它的定位，不往生产方案里塞。
